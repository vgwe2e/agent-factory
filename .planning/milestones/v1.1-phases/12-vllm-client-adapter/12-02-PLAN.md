Create `src/infra/backend-factory.ts` exporting:

```typescript
export type Backend = "ollama" | "vllm";

export interface VllmBackendOptions {
  vllmUrl: string;       // Required: base URL of vLLM server
  vllmModel?: string;    // Default: "Qwen/Qwen2.5-32B-Instruct"
}

export interface BackendConfig {
  chatFn: ChatFn;
  backend: Backend;
}
```

`createBackend(backend: Backend, options?: VllmBackendOptions): BackendConfig`

- For "ollama": return `{ chatFn: ollamaChat, backend: "ollama" }`. No validation needed -- Ollama handles schemas natively.
- For "vllm":
  1. Require `options.vllmUrl` (throw if missing)
  2. Call `validateScoringSchemas()` -- if `valid: false`, throw an Error with message: `"vLLM schema validation failed:\n"` followed by each error as `"  - {schema}: {reason}"`. This satisfies VLLM-04 (reject incompatible schemas before starting).
  3. Create ChatFn via `createVllmChatFn(options.vllmUrl, options.vllmModel ?? "Qwen/Qwen2.5-32B-Instruct")`
  4. Return `{ chatFn, backend: "vllm" }`

Note: This plan does NOT include cloud provisioning (Phase 14). The `--vllm-url` is provided directly by the user, pointing to an already-running vLLM instance. Cloud provisioning will inject the URL automatically in Phase 14.

Write tests in `src/infra/backend-factory.test.ts`. Mock `validateScoringSchemas` to test both pass and fail paths. Mock `createVllmChatFn` to verify it receives correct args. Test ollama path returns ollamaChat directly.

**Modify `src/cli.ts`:**

1. Add two new Commander options:
   - `--backend <backend>` with choices `["ollama", "vllm"]`, default `"ollama"`, description `"Scoring backend (ollama or vllm)"`
   - `--vllm-url <url>` with description `"vLLM server URL (required when --backend vllm)"`, no default

2. In the action handler, after the Ollama connectivity check block:
   - Import and call `createBackend(opts.backend, { vllmUrl: opts.vllmUrl })`
   - If backend is "vllm" and `opts.vllmUrl` is missing, print error and exit(1): `"Error: --vllm-url is required when --backend is vllm"`
   - If backend is "vllm", skip the Ollama connectivity check (it is irrelevant for vLLM)
   - Wrap `createBackend` in try/catch -- if it throws (schema validation failure), print the error message in RED and exit(1)
   - Pass `backendConfig.chatFn` to `runPipeline` via `PipelineOptions.chatFn`

3. When backend is "vllm", print in the === Pipeline === section:
   ```
   Backend:     vllm
   vLLM URL:    {url}
   ```
   When backend is "ollama" (or omitted), print:
   ```
   Backend:     ollama (local)
   ```

4. Update the version string from "v0.1.0" to "v1.1.0".

**Modify `src/pipeline/pipeline-runner.ts`:**

Minimal change only -- the pipeline already accepts `chatFn` via `PipelineOptions.chatFn` and defaults to `ollamaChat`. The only change needed:

1. When `options.chatFn` is provided (i.e., vLLM path), skip the ModelManager `ensureScoringModel()` and `unloadAll()` calls. The vLLM path does not use local Ollama models. Guard with:
   ```typescript
   const useLocalModels = !options.chatFn; // chatFn provided = external backend
   if (useLocalModels) {
     await modelManager.ensureScoringModel();
   }
   ```
   And at cleanup:
   ```typescript
   if (useLocalModels) {
     await modelManager.unloadAll();
   }
   ```

2. Similarly, skip the Ollama-specific model manager creation entirely when chatFn is provided, to avoid unnecessary Ollama connectivity issues when using vLLM.

Do NOT change the scoring loop itself -- it already uses `options.chatFn` via `scoreOneOpportunity(opp, l4s, company, knowledge, options.chatFn)`. The DI pattern means zero scoring code changes.
