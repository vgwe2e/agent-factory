  <name>Task 1: Semaphore and timeout primitives</name>
  <files>src/infra/semaphore.ts, src/infra/semaphore.test.ts, src/infra/timeout.ts, src/infra/timeout.test.ts</files>
  <behavior>
    Semaphore:
    - acquire() resolves immediately when slots available, queues when full
    - release() unblocks the next queued acquire()
    - run(fn) acquires, executes fn, releases even if fn throws
    - Concurrent tasks are bounded to N (e.g., semaphore(3) never runs more than 3 at once)
    - run() rejects with the fn's error after releasing the slot

    Timeout:
    - withTimeout(fn, ms) resolves with fn's result when fn completes within ms
    - withTimeout(fn, ms) rejects with TimeoutError when fn exceeds ms
    - TimeoutError has name "TimeoutError" and includes the timeout duration
    - AbortSignal is passed to fn so it can clean up (optional use by fn)
  </behavior>
  <action>
    Create src/infra/semaphore.ts:
    - Export class Semaphore with constructor(maxConcurrency: number)
    - Methods: acquire(): Promise<void>, release(): void, run<T>(fn: () => Promise<T>): Promise<T>
    - Internal queue of resolve callbacks. acquire() checks if running < max, if so increments and resolves, else pushes to queue. release() decrements, shifts queue and resolves next if any.
    - run() is the primary API: acquires, awaits fn(), releases in finally block.

    Create src/infra/timeout.ts:
    - Export class TimeoutError extends Error with name="TimeoutError" and timeoutMs property
    - Export function withTimeout<T>(fn: (signal: AbortSignal) => Promise<T>, timeoutMs: number): Promise<T>
    - Uses AbortController. Races fn(signal) against a setTimeout reject. Clears timer on resolve. Aborts controller on timeout.

    Write tests using node:test and assert/strict. Use setTimeout/Promise tricks to verify semaphore ordering and timeout behavior. No external dependencies.
  </action>

    - enqueue(entry) adds entry to in-memory checkpoint and schedules a debounced write (100ms)

    - flush() forces an immediate write of current state (called at pipeline end)
    - Write uses atomic rename pattern: write to .checkpoint.json.tmp, then rename to .checkpoint.json

  </behavior>

    - CheckpointWriter type: { enqueue: (entry: CheckpointEntry) => void; flush: () => void; checkpoint: Checkpoint }
    - enqueue() pushes entry to checkpoint.entries, sets a debounce timer (100ms). If timer already pending, no-op (coalesces writes).
    - When timer fires: write JSON to .checkpoint.json.tmp via writeFileSync, then renameSync to .checkpoint.json (atomic rename prevents corruption on crash).
    - flush() clears any pending timer and writes immediately via the same atomic rename path.
    - The checkpoint object is mutated in place so callers holding a reference see updates.
