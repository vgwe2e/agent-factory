  <name>Task 1: RunPod serverless cloud provider</name>
  <files>src/infra/cloud-provider.ts, src/infra/cloud-provider.test.ts</files>
  <behavior>
    - createCloudProvider(config) returns CloudProvider with provision/healthCheck/teardown methods
    - provision() creates a RunPod serverless endpoint via runpod-sdk, returns ProvisionedEndpoint with endpointId and baseUrl
    - provision() polls endpoint status until READY, throws if timeout exceeded (default 300s)
    - healthCheck(endpoint) fetches GET /v1/models on the endpoint proxy URL and returns true if 200
    - provision() calls healthCheck after endpoint is READY, polling until vLLM model is loaded (up to 180s)
    - teardown(endpoint) purges the RunPod serverless endpoint
    - teardown is idempotent -- calling twice does not throw
    - If RUNPOD_API_KEY is missing from config, provision() throws with clear message
    - Endpoint proxy URL follows RunPod pattern: https://api.runpod.ai/v2/{endpointId}/openai/v1
  </behavior>
  <action>
    Install runpod-sdk: `cd src && npm install runpod-sdk`

    Create src/infra/cloud-provider.ts:
    - Export interface CloudProviderConfig: { apiKey: string; gpuType?: string; model?: string; templateId?: string; maxProvisionTimeoutMs?: number; maxHealthTimeoutMs?: number }
    - Export interface ProvisionedEndpoint: { endpointId: string; baseUrl: string; provisionedAt: Date }
    - Export interface CloudProvider: { provision(): Promise<ProvisionedEndpoint>; healthCheck(endpoint: ProvisionedEndpoint): Promise<boolean>; teardown(endpoint: ProvisionedEndpoint): Promise<void> }
    - Export function createCloudProvider(config: CloudProviderConfig): CloudProvider

    Defaults:
    - gpuType: "NVIDIA H100 80GB HBM3" (note: actual gpuIds value for RunPod API needs validation -- may need "H100_80GB" or similar identifier)
    - model: "Qwen/Qwen2.5-32B-Instruct"
    - templateId: use RunPod's vLLM Quick Deploy serverless template (look up current template ID from RunPod docs during implementation -- the SDK may have a way to specify the vLLM worker image directly)
    - maxProvisionTimeoutMs: 300_000 (5 min)
    - maxHealthTimeoutMs: 180_000 (3 min)

    provision() implementation:
    1. Validate config.apiKey exists, throw if not
    2. Use runpod-sdk to create a serverless endpoint. The SDK API is: `const runpod = RunpodSdk(apiKey); const endpoint = await runpod.endpoint.create({...})` -- consult the SDK types during implementation for exact field names.
    3. Key configuration for endpoint creation: name (e.g. "aera-eval-{timestamp}"), gpuIds matching H100, workerImage or templateId for vLLM, model passed as env var or args to vLLM. Set idleTimeout to 60 (seconds) so the endpoint auto-scales to zero after scoring completes (defense against orphaned instances). Set maxWorkers to 1 (single H100).
    4. Poll endpoint status every 5s until it reports ready. Use exponential backoff: 5s, 5s, 10s, 10s, 15s... capped at 15s. Throw if maxProvisionTimeoutMs exceeded.
    5. Construct baseUrl: `https://api.runpod.ai/v2/${endpointId}/openai/v1` (RunPod's OpenAI-compatible proxy for vLLM serverless endpoints)
    6. Poll healthCheck (GET baseUrl + "/models") every 5s until 200 response. This confirms vLLM has loaded the model. Throw if maxHealthTimeoutMs exceeded.
    7. Return { endpointId, baseUrl, provisionedAt: new Date() }

    healthCheck() implementation:
    - GET `${endpoint.baseUrl}/models` with 10s timeout via AbortSignal.timeout
    - Return true if response.ok, false otherwise
    - Catch network errors and return false

    teardown() implementation:
    1. Call runpod-sdk to delete/purge the endpoint: `await runpod.endpoint.purge(endpointId)` or equivalent SDK method
    2. Catch and swallow errors (idempotent -- endpoint may already be gone)
    3. Log teardown success/failure

    IMPORTANT: The runpod-sdk API surface may differ from what's documented in research. During implementation, inspect the SDK types (`node_modules/runpod-sdk/dist/index.d.ts` or similar) to determine the exact method names and parameter shapes. The SDK is at v1.1.2. If the SDK does not support serverless endpoint creation directly, fall back to raw fetch against the RunPod REST API (POST https://api.runpod.ai/v2/endpoint with appropriate headers).

    - Use node:test and assert/strict

    - Test: provision() returns endpointId and baseUrl on success
    - Test: provision() throws when timeout exceeded (mock slow polling)
    - Test: healthCheck() returns true on 200, false on error
    - Test: teardown() succeeds and is idempotent (calling twice does not throw)
    - Test: baseUrl follows expected pattern
    - Do NOT make real API calls in tests -- mock all HTTP/SDK interactions

    - tracker.stop() records the stop timestamp
    - tracker.summary() returns CostSummary: { gpuSeconds, gpuHours (formatted string), estimatedCost (formatted string), ratePerHour }
    - gpuSeconds is wall-clock seconds between start and stop
    - estimatedCost = (gpuSeconds / 3600) * ratePerHour, formatted as "$X.XX"
    - gpuHours formatted as "Xh Ym Zs" human-readable
    - If stop() not called, summary() uses current time
    - Default ratePerHour: 5.58 (RunPod H100 serverless rate)
    - start() can be called with an explicit Date for testability
  </behavior>

    - Export function createCostTracker(ratePerHour?: number): CostTracker
