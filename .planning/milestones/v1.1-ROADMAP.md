## Phase Details

### Phase 12: vLLM Client Adapter
**Goal**: Users can score opportunities through a vLLM backend using the same interface and CLI as the existing Ollama path
**Depends on**: Phase 11 (v1.0 complete)
**Requirements**: VLLM-01, VLLM-02, VLLM-03, VLLM-04
**Success Criteria** (what must be TRUE):
  1. User can run `aera-evaluate --backend vllm --input export.json` and get scored output identical in structure to the Ollama path
  2. User can omit the `--backend` flag and the engine defaults to Ollama with no behavioral change from v1.0
  3. Engine rejects incompatible scoring schemas before starting a vLLM run, reporting which schema failed and why
  4. Engine translates all existing Zod scoring schemas to vLLM `response_format` automatically -- user changes nothing about their workflow
**Plans:** 2/2 plans complete

Plans:
- [ ] 12-01-PLAN.md -- vLLM client adapter and schema translation layer
- [ ] 12-02-PLAN.md -- Backend factory, CLI flags, and pipeline wiring

### Phase 13: Concurrent Pipeline Runner
**Goal**: Engine scores multiple opportunities simultaneously with bounded concurrency, safe checkpointing, and visible progress
**Depends on**: Phase 12
**Requirements**: CONC-01, CONC-02, CONC-03, CONC-04
**Success Criteria** (what must be TRUE):
  1. User can configure concurrency (e.g., `--concurrency 15`) and the engine scores that many opportunities in parallel
  2. If the engine crashes mid-run and restarts, no checkpoint data is lost or corrupted -- previously scored opportunities are skipped on resume
  3. A stuck vLLM request times out and releases its semaphore slot without blocking the remaining queue
  4. User sees a live progress summary showing in-flight count, completed count, error count, and estimated time remaining
**Plans:** 2/2 plans complete

Plans:
- [ ] 13-01-PLAN.md -- Concurrency primitives (semaphore, timeout, concurrent-safe checkpoint)
- [ ] 13-02-PLAN.md -- Pipeline integration, progress display, and --concurrency CLI flag

### Phase 14: Cloud Infrastructure
**Goal**: Engine can provision, use, and tear down an ephemeral H100 GPU endpoint automatically -- user only provides a RunPod API key
**Depends on**: Phase 12
**Requirements**: CLOUD-01, CLOUD-02, CLOUD-03
**Success Criteria** (what must be TRUE):
  1. User runs `aera-evaluate --backend vllm --input export.json` with a RunPod API key set, and the engine provisions an H100 endpoint, waits for it to be healthy, scores all opportunities, and tears it down -- no manual cloud console interaction
  2. If the user hits Ctrl+C or the process crashes, the cloud instance is torn down within 60 seconds (defense-in-depth: signal handlers + instance TTL)
  3. After a run completes, the engine reports GPU-hours consumed and estimated dollar cost in the terminal output and in the evaluation artifacts
**Plans:** 3/3 plans complete

Plans:
- [x] 14-01-PLAN.md -- RunPod serverless cloud provider and GPU cost tracker
- [x] 14-02-PLAN.md -- CLI wiring, signal handler teardown, and pipeline cost integration
- [x] 14-03-PLAN.md -- Gap closure: write cost artifact to evaluation directory, fix health timeout
