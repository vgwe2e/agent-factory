<objective>
Refactor the Aera Feasibility Engine to score at the **skill level** (826 skills nested under `hierarchy[].skills[]`) instead of the current L3 opportunity level (362 `l3_opportunities[]` rollups).

The L3 `opportunity_name` values (e.g., "Autonomous MRP-Driven Production & Supply Alignment Engine") are synthetic rollup labels generated by the Aera platform — they don't exist in the application UI and can't be reconciled by users. The actual items users see and interact with in the Aera app are the **skills** nested at `hierarchy[].skills[]` (e.g., "BOM Release Optimization", "Constraint-Aware BOM Optimization"). These are the entities that must be scored, reported on, and presented in the PowerPoint.

The skills also carry far richer data than L3 rollups — each skill has `problem_statement`, `execution`, `actions`, `constraints`, `archetype`, `max_value`, `loe`, `differentiation`, and `aera_skill_pattern` — all of which should be leveraged by the scoring prompts for more accurate assessments.
</objective>

<context>
Read the CLAUDE.md at the repo root for full architecture and conventions.

The current pipeline flow is:
1. **Ingestion** (`src/ingestion/parse-export.ts`) → validates JSON, returns `HierarchyExport`
2. **Triage** (`src/triage/`) → red-flag detection + tier assignment on L4 activities grouped by L3
3. **Scoring** (`src/scoring/scoring-pipeline.ts`) → iterates `l3_opportunities[]`, scores each L3 across 3 lenses (technical, adoption, value)
4. **Simulation** (`src/simulation/`) → generates component maps, decision flows, etc. for promoted opportunities
5. **Output** (`src/output/`) → TSV, reports, PPTX generation

The JSON export structure has three levels:
- `project.l3_opportunities[]` — 362 items, aggregated rollups with synthetic `opportunity_name` (CURRENTLY SCORED)
- `project.hierarchy[]` — 2,016 L4 activities, each with `l1`, `l2`, `l3`, `name`, `description`, `financial_rating`, `ai_suitability`, etc.
- `project.hierarchy[].skills[]` — 826 total skills, the actual Aera app entities (NOT CURRENTLY SCORED)

Each skill object has these fields:
```
id, name, description, archetype, max_value, slider_percent, overlap_group,
value_metric, decision_made, aera_skill_pattern, is_actual, source, loe,
savings_type, actions[], constraints[], execution{}, program_focus_ids[],
problem_statement{}, differentiation, generated_at, prompt_version,
is_cross_functional, cross_functional_scope, operational_flow[],
walkthrough_decision, walkthrough_actions[], walkthrough_narrative
```

The `problem_statement` object contains: `current_state`, `quantified_pain`, `root_cause`, `falsifiability_check`, `outcome`.

The `execution` object contains: `target_systems`, `write_back_actions`, `execution_trigger`, `execution_frequency`, `autonomy_level`, `approval_required`, `approval_threshold`, `rollback_strategy`.

Key files to read before making changes:
- `src/schemas/hierarchy.ts` — Zod schemas, `l4ActivitySchema` currently has `skills: z.array(z.unknown())`
- `src/types/hierarchy.ts` — TypeScript types inferred from schemas
- `src/types/scoring.ts` — `ScoringResult` interface, currently keyed on `l3Name`
- `src/scoring/scoring-pipeline.ts` — `scoreOneOpportunity()` takes `L3Opportunity` + `L4Activity[]`
- `src/scoring/prompts/technical.ts`, `adoption.ts`, `value.ts` — the LLM prompt templates
- `src/triage/` — red flag detection and tier assignment
- `src/output/format-scores-tsv.ts` — TSV output columns
- `src/output/write-final-reports.ts` — report generation
- `src/pipeline/pipeline-runner.ts` — top-level orchestrator
- `src/pipeline/scoring-to-simulation.ts` — bridges scoring → simulation
- `src/cli.ts` — CLI entry point
- `src/generate_pptx.py` — PowerPoint generator (Python)
</context>

<requirements>

<schema_changes>
1. **Define a proper Zod schema for skills** — replace `skills: z.array(z.unknown())` in `l4ActivitySchema` with a typed `skillSchema` that validates the full skill object structure (id, name, description, archetype, max_value, problem_statement, execution, actions, constraints, etc.)

2. **Create a `Skill` TypeScript type** inferred from the new schema, exported from `types/hierarchy.ts`

3. **Add a `SkillWithContext` type** that enriches a skill with its parent L4 activity context (l1, l2, l3, l4_name, financial_rating, ai_suitability) — the scoring prompts need this hierarchy context
</schema_changes>

<scoring_changes>
4. **Refactor `ScoringResult`** — replace `l3Name`/`opportunityName` with skill-level identifiers:
   - `skillId: string` — the skill's `id` from the JSON
   - `skillName: string` — the skill's `name` (this is the reconcilable name users see in the Aera app)
   - `l4Name: string` — parent L4 activity name
   - `l3Name: string` — L3 category (kept for grouping/reporting)
   - `l2Name: string`, `l1Name: string` — kept for hierarchy breadcrumbs
   - `archetype: LeadArchetype` — from the skill's own `archetype` field (no longer inferred)
   - Remove `archetypeSource` since skills carry their own archetype

5. **Refactor `scoreOneOpportunity` → `scoreOneSkill`** — accept a `SkillWithContext` instead of `L3Opportunity` + `L4Activity[]`. The skill already has `archetype`, `max_value`, `problem_statement`, `execution`, `actions`, `constraints`, and `differentiation` — use all of this in the scoring prompts.

6. **Refactor `scoreOpportunities` generator** — iterate over all skills extracted from the hierarchy instead of iterating L3 opportunities. Group/sort by parent L3 for progress tracking.

7. **Update the three scoring prompts** (`technical.ts`, `adoption.ts`, `value.ts`) to leverage skill-level data:
   - **Technical**: Use `execution.target_systems`, `aera_skill_pattern`, `actions[]`, `constraints[]` for more precise platform fit assessment
   - **Adoption**: Use `execution.autonomy_level`, `execution.approval_required`, `loe`, `problem_statement.current_state` for realistic adoption scoring
   - **Value**: Use `max_value`, `value_metric`, `savings_type`, `problem_statement.quantified_pain` for grounded value assessment
</scoring_changes>

<triage_changes>
8. **Refactor triage to operate at skill level** — red flags and tier assignment should evaluate individual skills, not L3 rollups. Skills without `problem_statement` or with `max_value === 0` are natural red flag candidates.
</triage_changes>

<output_changes>
9. **Update TSV output** (`format-scores-tsv.ts`) — columns should include `skill_id`, `skill_name`, `l4_name`, `l3_name`, `l2_name`, `l1_name`, `archetype`, and all existing score columns. Multiple skills sharing the same parent L4 should each be their own row — the relationship is many skills to one L4 activity.

10. **Update all report formatters** to reference skill names instead of L3 opportunity names. When grouping for summaries, group by parent L4 activity, then by L3.

11. **Update `generate_pptx.py`** — the top-10 selection should rank by individual skill composite score, not L3 rollup. Each slide's primary title is the `skill_name` (the exact name from the Aera app). Include `l4_name > l3_name` in the hierarchy breadcrumb. Use the skill's own `problem_statement` fields directly — do not use L3-level `opportunity_summary` or `rationale`.
</output_changes>

<pipeline_changes>
12. **Update `pipeline-runner.ts`** — the orchestrator needs to extract skills from the hierarchy, attach parent L4 context, and pass `SkillWithContext[]` through the pipeline

13. **Update checkpoint system** — checkpoints should key on `skill_id` instead of `l3_name`

14. **Update `scoring-to-simulation.ts`** — bridge should pass skill-level data to simulation
</pipeline_changes>

</requirements>

<constraints>
- **Do not remove L3/L4 data from the schema** — keep the full hierarchy structure; we're adding skill-level scoring, and L3/L4 context is still needed for grouping and reporting
- **The `l3_opportunities` array can be ignored by the scoring pipeline** — it's a pre-computed rollup from the Aera platform that we no longer need to score. Don't delete it from the schema (it's still in the JSON) but the pipeline should not consume it.
- **Maintain the never-throw pattern** — all LLM calls return `ChatResult` union types
- **Maintain TDD approach** — update existing tests first to match new interfaces, then implement. Co-located test files.
- **Skill `archetype` is authoritative** — don't re-infer archetypes; each skill already declares its own (`DETERMINISTIC`, `AGENTIC`, or `GENERATIVE`)
- **Some hierarchy entries have 0 skills** — 1,362 of 2,016 L4 activities have no skills. These should be skipped gracefully.
- **Skills with `is_actual: true`** may need different treatment — investigate and handle appropriately
- **Preserve the `--max-tier` CLI flag** — tier assignment still applies, just at skill level now
- **Backend agnostic** — changes must work with both Ollama and vLLM backends
</constraints>
